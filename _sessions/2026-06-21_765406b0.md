## How pure-Python (Streamlit) works

Streamlit = you write one linear Python script. No HTML/CSS/JS, no API, no build step. Every widget is one function call that returns a value:

```python
import streamlit as st
import plotly.express as px

probs = load_parquet("predictions.parquet")        # your export
ts   = st.slider("timestep", min, max)              # returns the picked time
hzn  = st.selectbox("horizon", ["30m","1h","3h","6h"])
model= st.radio("model", ["B","A"])

frame = probs.query("ts_utc == @ts")[[hzn, model]]  # 27 oblasts, one value each
fig   = px.choropleth(frame, geojson=ua_adm1, color=hzn)  # map
st.plotly_chart(fig)
st.dataframe(metrics)                                # stats menu
```

Mechanics: Streamlit **re-runs the whole script top-to-bottom on every widget change**. Move slider → script reruns → new `ts` → new map. State lives in the rerun. That's the whole model. `@st.cache_data` stops it reloading the parquet each rerun.

## Why people say it "sucks"

Real, specific failure modes — not vague:

- **Rerun model bites you.** Anything slow not wrapped in `@st.cache_data` re-executes on every click → laggy. Fixable, but it's the #1 gotcha.
- **Layout is rigid.** Top-to-bottom flow + columns. No fine control. Looks like a dashboard, not a product. Fine for demo, not portfolio-polished.
- **Choropleth interactivity is shallow.** Plotly map in Streamlit = hover + color. Click-an-oblast-to-drill-down needs `streamlit-plotly-events` (extra dep, finicky) or a Folium component. This is where Streamlit gets annoying for *your* spec (per-region click → detail).
- **Animation/time-scrubbing** is a slider rerun, not smooth playback. Fine, but not slick.

When it's "easier to just go React": when you want click-to-drill, custom layout, smooth animation, deploy as a static site. Your spec (click region → its stats, A-vs-B side panel) leans toward React's strengths.

## My confidence — honest

**Streamlit faster for me to build: ~80% confident.** Where that confidence holds vs cracks:

- Solid for: choropleth colored by prob, slider, horizon/model selectbox, metrics table. That's 90% of your ask, and it's genuinely ~half a day, low risk. Plotly ships a maps stack; I wire data → done.
- **The crack:** Ukraine ADM1 geojson must have `id`/property keys that *exactly* match your 27 oblast slugs (`kyiv-city`, `crimea`...). Mismatch = blank map. This bites **both** Streamlit and React equally — it's a data-join problem, not a stack problem. ~1h either way to reconcile.
- Where Streamlit could turn into a slog: if you want polished click-drilldown + nice layout, I'd be fighting the framework, and the time gap vs React narrows fast.

**React honesty:** more total code (export step + SPA + map lib + styling), but every piece is predictable for a coding agent — no framework surprises, you already run a React frontend. Slower to first-pixel, but no "Streamlit won't let me" walls. ~1.5–2 days.

## Recommendation

Given **static export first** (you chose it) and your spec has click-region-drilldown: do the **export as step one regardless** — it's identical work for either frontend. Then build Streamlit as the throwaway-fast demo. If it feels too limited when you click around, the export already exists → React is purely additive, nothing wasted.

So the data layer is the real first task and it's stack-agnostic. That de-risks the whole decision.
