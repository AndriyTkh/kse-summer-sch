### 🤖 Claude

**Isotonic calibrator = function that fixes predicted probabilities so they match reality, without changing the ranking.**

## Problem it solves
Model outputs a "probability" 0.8. But of all rows it called 0.8, maybe only 60% actually had alerts. Probs are **miscalibrated** — overconfident. PR-AUC doesn't care (ranking still fine), but if you act on the number ("80% → evacuate"), the number must be true. That's calibration.

## What isotonic does
Learns a mapping `raw_prob → calibrated_prob` from held-out data (your **calib** fold):
- Bin/sort predictions by raw score.
- For each, measure actual observed alert frequency.
- Fit a curve: x = raw prob, y = observed freq.

**Constraint: the curve can only go up (monotone non-decreasing).** "Isotonic" = order-preserving. Higher raw score → higher (or equal) calibrated score, never lower.

```
observed
freq  1.0|                    ____----
         |              _____/
      0.5|         ____/   <- isotonic fit (step/stair shape)
         |    ____/
      0.0|___/________________________
         0.0      raw prob        1.0
```

## Two key consequences
1. **Ranking unchanged → PR-AUC unchanged.** Monotone map keeps order. That's why your PR-AUC identical raw vs calibrated, but ECE/Brier dropped.
2. **Non-parametric** — no fixed shape (unlike Platt/sigmoid). Fits any monotone distortion. Cost: needs enough calib data or it overfits to step noise.

## In your code
[evaluate.py](src/evaluate.py) `fit_isotonic` wraps sklearn `IsotonicRegression(out_of_bounds="clip")`:
- Fit on **calib** fold only (out-of-fold → honest, no leak).
- `clip` = test probs outside calib's seen range get clamped to the end values, not extrapolated.
- One calibrator per horizon ([run_mvp.py:51-54](run_mvp.py#L51-L54)).

That's the 0.084 → 0.013 ECE drop: raw 0.8 that meant 0.6 → isotonic rewrites it to ~0.6.

**Isotonic vs Platt:** Platt fits a sigmoid (2 params, smooth, good for small data). Isotonic fits a free staircase (flexible, needs more data). You have ~4 weeks calib × 27 oblasts = plenty → isotonic right pick.

## 1. Probability as a frequency

When the model says "0.7" for an oblast-hour, it means: *across many situations that look like this one, an alert happens about 70% of the time.* It is not a yes/no. It's a long-run rate attached to a single row.

Two things can be true or false independently about that 0.7:
- **Does it rank correctly?** When the true probability is higher, is the predicted number higher? (called *discrimination*)
- **Is the number itself right?** Of all rows scored 0.7, do ~70% actually fire? (called *calibration*)

Almost every metric in your table measures one or the other. Keep these two ideas separate — most confusion comes from mixing them.

## 2. The target is an event, not a thing

Each row is one (oblast, hour). The thing you predict — the **target** or **label** — is a yes/no event:

> "Will an alert occur somewhere in the window from now (t) to t+H?"

H is the horizon. So the *same hour* gets four different labels — one per horizon — because the window is wider each time. This is the seed of the whole table.

## 3. Base rate

The **base rate** is just the fraction of rows where the label is 1 (the event happened) in your test data.

base 0.315 means 31.5% of oblast-hours had an alert in the next window.

Why it rises 0.315 → 0.458 as H grows: a wider window is easier to "catch" an alert. In the next 30 minutes, probably nothing. In the next 6 hours, much more likely *something* happens. Bigger net, more fish. The base rate is the **prevalence** of the positive class, and it's the floor every other score is measured against.

## 4. Why not accuracy

Beginners reach for accuracy (% of rows guessed right). On rare events it lies.

Imagine base rate 5%. A model that *always says "no alert"* is 95% accurate and completely useless. Accuracy rewards predicting the common answer. Your data is **imbalanced** (rare positives), so accuracy is banned here. This is issue #5 in the plan, and it's why the table has no accuracy column.

## 5. Precision and recall

Instead, we judge the model on how it handles the *positives*, using two numbers. Pick a threshold (say "alarm if prob > 0.6"):

- **Precision** = of all the times you raised an alarm, what fraction were real? (Did you cry wolf?)
- **Recall** = of all the real alerts, what fraction did you catch? (Did you miss any?)

These trade off. Raise the threshold → fewer alarms, each more likely right (precision up) but you miss more (recall down). Lower it → catch everything (recall up) but lots of false alarms (precision down). There is no single threshold that's "correct" — it depends how much a miss costs vs a false alarm.

## 6. PR-AUC (the headline metric)

Because no single threshold is right, you sweep *every* threshold from 0 to 1, plot precision against recall at each, and get a curve. The **area under that curve** is **PR-AUC** (also called average precision).

- 1.0 = perfect: there's a threshold ordering where you catch all alerts with no false alarms.
- A useless model that ranks randomly scores PR-AUC ≈ the base rate.

That second point is the key. **PR-AUC's floor is the base rate, not 0.** A PR-AUC of 0.315 on data with base 0.315 means the model learned nothing. Your 0.901 against a floor of 0.315 means it ranks risky oblast-hours far above the floor.

PR-AUC only cares about **ranking** (concept 1, discrimination). It does not care whether 0.7 literally means 70%. That's why calibration later changes the probabilities but leaves PR-AUC untouched.

## 7. Lift

**lift = PR-AUC ÷ base rate.** "How many times better than the floor."

This is why lift falls (2.86 → 2.01) while PR-AUC rises (0.901 → 0.919): the floor rises faster than the score. At 6h the event is common (base 0.458), so even a good ranking can't be *many times* better than "it's usually happening anyway." At 30m the event is rare, so a good ranking is a much bigger relative achievement.

Both readings are correct and they tell the project's story: **the model's value is greatest at short horizons**, exactly where a baseline struggles and where early warning matters most.

## 8. Calibration — ECE

Now switch from ranking to *are the numbers honest* (concept 1, calibration).

To measure it: bucket all predictions by their value (all the ~0.1s, all the ~0.2s, …). In each bucket compare the **average predicted probability** to the **actual fraction that fired**. If the model says 0.8 for a bucket but only 0.6 fired, it's overconfident by 0.2 there.

**ECE (Expected Calibration Error)** is the average of those gaps across buckets, weighted by bucket size. 0 = perfectly honest probabilities; higher = more lying.

- ECE0 = before fixing (raw model).
- ECE = after the isotonic calibrator (the function I described last turn) rewrites the probabilities to match observed frequencies.

Your 0.084 → 0.013 at 30m means the raw model was meaningfully overconfident and the calibrator fixed it. At 6h it was already nearly honest (0.049), so little changed. Calibration is a *separate* skill from ranking — a model can rank perfectly and still quote dishonest numbers, or vice versa.

## 9. Brier score

**Brier = average of (predicted probability − actual outcome)², where outcome is 0 or 1.** It's just mean squared error for probabilities. Lower is better.

It's a *combined* score: it punishes both bad ranking and bad calibration at once. That's why it moves less dramatically than ECE — calibration fixes only part of what Brier measures. At 6h, Brier flat (0.110 → 0.110) because there was almost no calibration error to remove and ranking was untouched.

Why keep both ECE and Brier? ECE isolates calibration; Brier gives an overall single-number sanity check. They agree here, which is reassuring.

## 10. The thread tying it together

- **Discrimination** (does it rank?) → measured by PR-AUC and lift → strong, ~0.90+, best relative value short-horizon.
- **Calibration** (are the probs honest?) → measured by ECE → fixed by isotonic, now ≤0.04.
- **Both at once** → Brier → low.
- All measured **only on held-out future data**, with a purge gap so no label leaks across the train/test cut → the numbers reflect genuine forecasting, not memorized history (this is the leak guard, concept underlying everything: a metric is only meaningful if the test data was never used to build or tune the model).

That last point is the whole reason the plan is so strict about temporal splits and purging. A great-looking PR-AUC means nothing if the model peeked at the answers.

## Two different "future" things

**Backtest (what the table measures):** pick a past cut, predict t→t+6h, and the true labels already exist so you can score. Crucially, every feature row before the cut is **fully populated** — the historical CSV has all the wave/missile data filled in retroactively.

**Live forecast (the actual product):** predict the next 6h from *now*. Two things break that the backtest never sees:

1. **No labels yet.** The thing you're predicting hasn't happened → you can't compute PR-AUC at prediction time. You only learn if you were right 6h later. So scoring it is a *delayed, rolling* process, not a one-shot number.

2. **Incomplete features at the right edge — the key point you're making.** Leak guard says "features at t use data < t." But live, data < t hasn't all *arrived* yet. The launch/wave/missile sources publish with lag (hours to a day). So the most recent rows — exactly the ones that matter most for a 6h forecast — are missing or partial in live inference, but were **complete** in the backtest.

That second one means **backtest PR-AUC overstates live performance.** Not because of leakage (the guard is correct on *timestamps*), but because of *data availability* — the backtest assumes you have data you won't actually have yet when forecasting live. This is the "right-edge / ragged-edge" problem in nowcasting.

## So yes — separate metric, and it must simulate latency

A live-honest eval can't just reuse the clean grid. It has to:
- For each forecast origin, **mask each feature source to what would actually have arrived by then** (apply each source's real publish lag), then predict.
- Score those forecasts against labels that land later.
- Report this *separately* from the backtest — call it operational/forward PR-AUC. The gap between backtest and operational numbers is itself a headline finding ("model scores 0.90 backtest, 0.Xx live").

This is exactly the kind of thing that's "a large point to check" like you said — the difference between the two is the honest live capability.
