## Solution branches

### A. Classical statistical
**Stack:** `pandas`, `statsmodels`, `pmdarima`

- **ARIMA [AutoRegressive Integrated Moving Average]** — models series as function of own past values (AR) + past errors (MA), differencing (I) removes trend. Good baseline, needs stationarity.
- **SARIMA [Seasonal ARIMA]** — adds seasonal terms. Air raids have strong daily/weekly cycle → seasonal fits.
- **Prophet** (Meta lib) — decompose into trend + seasonality + holidays. Easy, robust to gaps/outliers. Few params.

**Why start here:** low skill barrier, fast, strong baseline. Beats fancy models often on small data.

### B. Machine learning (tabular)
**Stack:** `scikit-learn`, `XGBoost` / `LightGBM`

- Turn time series → supervised table via **lag features** (value at t-1, t-2…) + calendar features.
- **Gradient boosting** — ensemble of decision trees, each tree fixes previous tree's errors. Handles nonlinearity, mixed features well.
- Predict count/duration per window.

**Why:** mid skill, very strong on tabular, handles external features easy. Recommend as main if forecasting.

### C. Deep learning (sequence)
**Stack:** `PyTorch` / `Keras`, `pandas`

Recurrent nets — read sequence step by step, keep hidden "memory" state:
- **RNN [Recurrent Neural Network]** — base. Feeds output back as input. Suffers vanishing gradient → forgets long past.
- **LSTM [Long Short-Term Memory]** — adds gates (input/forget/output) + cell state. Gates decide what to keep/drop → learns long dependencies. Heavier.
- **GRU [Gated Recurrent Unit]** — LSTM simplified: 2 gates (reset/update), no separate cell state. Fewer params, trains faster, ~same accuracy. **Pick GRU over LSTM** on small data/2-day budget.

**LSTM vs GRU short:** GRU = lighter, faster, less data needed. LSTM = more expressive, more params, risk overfit on small set.

**Why:** high skill barrier + needs lots data. Risky for 2 days. Do only as stretch/comparison.

### D. Point process / event models
**Stack:** `tick`, custom

- **Hawkes process** — self-exciting: each event raises prob of near-future events. Models attack waves (one alert triggers cascade). Conceptually perfect fit, but niche/hard.

**Why:** elegant but skill+time risk high. Mention in report, skip impl.

### E. Anomaly detection branch
**Stack:** `scikit-learn`, `statsmodels`

- **STL [Seasonal-Trend decomposition using Loess]** + residual threshold — flag points far from expected.
- **Isolation Forest** — isolates outliers by random splits; anomalies need fewer splits.

## Why each caps there

**B — short, sharp.** Lag features are the power AND the limit. To predict t+6 you need lags up to t. Two modes:
- **Direct multi-step:** train separate model per horizon (model_6h, model_24h). Acc drops fast each step.
- **Recursive:** feed prediction back as input → error compounds, blows up by ~24h.
- Past lag-window's reach (~48h) B just returns climatology = "average Tuesday 3am." Not forecast.

**C — slightly further.** Hidden state carries longer pattern (wave structure, escalation tempo) → stretches to ~12–48h before drift dominates. Why C earns its place: this is its one real edge over B.

**A — different kind.** Doesn't "run out" — Prophet predicts *months* ahead. BUT only the **average seasonal shape** ("nights worse than days, ~X alerts/week"). Zero skill on *which* hour. So infinite horizon, near-zero point-accuracy long-out. Useful as **baseline rate**, not event prediction.

## The real split — two questions, two horizons

1. **"Alert in next N hours?"** (event timing) → B/C, horizon **hours**. Sharp, actionable (shelter/resource).
2. **"How many alerts next week/month?"** (aggregate load) → A/Prophet/monthly-count, horizon **weeks**. Trend/seasonal, planning-level, no specific-time skill.

Pick which your project answers. Most defense value = **#1, short-horizon, high-precision**. Recommend frame project as "next 6–12h alert probability per oblast." Honest, achievable, useful.

## #3 Quantile LightGBM — what it takes

Normal B predicts **one number** (mean/expected). Quantile = predicts a **range**.

How: LightGBM has `objective='quantile'` + `alpha` param. Train 3 models:
- `alpha=0.1` → 10th percentile (low bound)
- `alpha=0.5` → median
- `alpha=0.9` → 90th percentile (high bound)

→ output "alert prob 60–80%, mid 70%." Gives the uncertainty band A has free, B lacks.

Cost: ~2h (just retrain same features, 3 alphas per horizon). Why not v1: multiplies model count (3 × 4 horizons = 12), adds eval complexity. Defer. Note in roadmap.

## The real problem: train resolution ≠ serve resolution

Core ML truth you spotted:

> **A model can only use a feature at the resolution it was trained on.**

Train with `drones_last_night` (daily) → model learned a **daily relationship**. Feed it a precise live hourly count at inference → it doesn't know what to do. The weight maps daily-number→outcome. Hourly number = different distribution → garbage or no gain.

**You can't bolt finer input onto a coarser-trained model.** To consume precise data, the model must have *seen precise data in training*. And that needs **precise history** — which for drones you only have daily (AF) or event-level (CSIS), not the real-time per-oblast stream.

So three real paths:

### 1. Match resolution — train at finest history you actually have
If CSIS gives event-level → resample BOTH train + serve to that grid (e.g. hourly). Model learns hourly drone→alert relation, consumes hourly at inference. **Consistency is the rule: same resolution train and serve.** Best clean upgrade.

### 2. Hybrid: daily ML base + real-time rule layer (no retrain)
Keep daily model for baseline. Add a **separate live adjustment** outside it:
- live monitoring count spikes → rule bumps alert-prob / overrides high.
- ML handles pattern, rule handles fresh magnitude.
- This is how you exploit real-time data **without** finer history. Deployable, honest. (Same hybrid as MiG-override.)

### 3. Two-model / nowcasting tier
- **Long model:** daily/6h, trained on daily features (have history).
- **Nowcast model:** short-horizon (30min–2h), trained ONLY on scraped per-wave history, takes live partial count → updates prediction mid-attack.
- Each trained at its own resolution. Switch/blend by horizon.

### Why you can't cheat it
- **Disaggregating** daily→hourly by a fixed profile adds *no information* — you're inventing the shape. Model gains nothing real.
- **Upsampling** live data to daily to match the model = throws away the precision you wanted.
