## The cost model

Feature count here ≈ **60 cols** (9 lag/recency + 5 calendar + **42 threat** + 4 tempo).

Three cost centers in the walk:

| Center | Cost ∝ | Driven by |
|---|---|---|
| **LGBM refits** (dominant) | rows × **features** × trees, **× retrain_count** | iteration/retrain count × feature count |
| **PSI scan** (every block) | **features** × block_rows, **× blocks** | blocks × feature count |
| per-block predict | rows × trees | blocks |

So feature count and block/iteration count appear **multiplied** in both heavy terms, not summed.

## Which lever is bigger

- **Feature count = linear per-fit multiplier.** 60→30 features ≈ ~2× faster *per* LGBM fit and per PSI scan. Clean but flat.
- **Iteration/retrain count = count of how many expensive fits happen.** `block_days` 14→28 ≈ halves blocks → halves PSI scans **and** halves periodic/drift retrain opportunities → fewer of the heavy fits. Compounds.

A single LGBM fit (~116k rows × 60 feats × hundreds of trees) dwarfs a single PSI scan (60 feats × ~9k rows = histogram, trivial). So total wall time ≈ **(number of refits) × (rows × features × trees)**. The *number of refits* is set by iteration/retrain count; feature count just scales each one.

**Verdict:** trimming features gives a flat ~linear win on every fit; trimming iterations cuts the *number* of fits — the stronger lever. They're not comparable as add-on costs; they're factors in the same product.

### 🤖 Claude

Walk-forward is **two different things** people conflate — and that's the whole confusion. Let me split them.

## 1. Walk-forward as EVALUATION (robustness check)

The single holdout gives **one verdict**: "0.92 PR-AUC on the last 8 weeks." But — is that a lucky window? Does it hold in winter vs summer, 2023 vs 2025? You can't tell from one split.

Walk-forward eval slides the train/test cut forward N times → N folds → **mean ± spread** across war regimes. 

**What it gives:** a *confidence* number. "0.92 ± 0.01 across 4 regimes" is a credibility claim; "0.92 on one window" is a hope. This is a **measurement tool** — it doesn't change the model, it audits it.

## 2. Walk-forward as RETRAINING (the drift harness, `retrain.py`)

Walk the model forward block-by-block; when drift fires, retrain on a recent window. 

**What it gives — three things:**
- **Staleness cost:** how much a frozen 2022 model decays vs one that keeps adapting (frozen vs drift vs periodic).
- **When-to-retrain signal:** the PSI drift trigger tells you *the world changed enough, refit now* — so in production you're not retraining blindly on a calendar.
- **The production protocol itself:** in deployment you *cannot* train once and run forever — new data arrives, the war shifts. Walk-forward is the template for "retrain as you go."

## What OUR experiment actually found

Remember the single-fit-vs-walkforward test: **adaptive gain ≤ 0.005 PR-AUC**. Why so small?
- B already has **recency weighting** (exponential half-life) → it down-weights stale data *within* a single fit.
- A recent single fit already sits in the current regime.

So the retraining **adaptation** buys almost nothing here — recency weighting eats its lunch.

## So what's it worth in *this* project?

| | value here | role |
|---|---|---|
| WF **eval** (variance) | **real** — turns "0.92 on one window" into "0.92±σ across regimes" | keep as a **credibility study** for the KSE submission |
| WF **retrain** (adaptation) | **small** (≤0.005) — recency weighting already handles drift | demote to **run-once finding** + production template |
| WF **drift trigger** | **real operationally** — tells you *when* to refit live | keep as the **deployment scaffolding** |

The finding itself is a result worth reporting: **"adaptive retraining barely beats a recent single fit ⇒ the regime drifts slowly and recency weighting suffices."** That's *why* the cheap production path (single recent fit) is justified — the walk-forward is what *proves* you don't need the expensive path.

So: it's not an accuracy booster here. It's the **honesty audit** (eval) + the **proof the cheap path is safe** (retrain finding) + the **drift alarm for live ops**. Keep it as a study, not the hot path — exactly the earlier recommendation, now with the "why."
